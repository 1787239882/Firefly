# 📘 Part 1：数据仓库四层架构

## 1.1 ODS 原始数据层

> ODS = Operational Data Store，操作数据存储

**一句话**：把源系统的数据原封不动搬过来，不做任何加工。

**你在银行看到的内容**：
- 核心银行系统的账户交易流水
- 信贷系统的贷款申请记录
- 柜面系统的操作日志

**特点**：
- 结构和源系统一模一样
- 数据最完整、最原始
- 一般不直接对外提供 API

**工作中你要知道**：
当你接到需求说"这个接口要取原始交易流水"——大概率是从 ODS 或 DWD 取数。

---

## 1.2 DWD 明细数据层

> DWD = Data Warehouse Detail，数据仓库明细层

**一句话**：把 ODS 数据做清洗、标准化，去掉脏数据。

**做了什么加工**：
- 统一编码：源系统 A 用 `M`/`F` 表示性别，源系统 B 用 `0`/`1`
- 去重去噪：去掉测试数据、空值处理
- 维度关联：把客户 ID 关联上客户维度表

**工作中你要知道**：
大部分查询接口从 DWD 层取数，数据干净、结构标准。

---

## 1.3 DWS 汇总数据层

> DWS = Data Warehouse Summary，数据仓库汇总层

**一句话**：把明细数据按维度汇总起来。

**典型例子**：
```sql
-- 从 DWD 的逐笔交易表，汇总成 DWS 的每日汇总表
INSERT INTO dws.customer_daily_summary
SELECT
    customer_id,
    DATE(trans_time) AS trans_date,
    COUNT(*)        AS trans_count,     -- 交易笔数
    SUM(amount)     AS total_amount,    -- 交易总额
    AVG(amount)     AS avg_amount       -- 平均金额
FROM dwd.transaction_detail
WHERE DATE(trans_time) = '2025-06-15'
GROUP BY customer_id, DATE(trans_time);
```

**工作中你要知道**：
报表类接口、统计类接口经常从 DWS 取数，**速度比从 DWD 快 10 倍**。

---

## 1.4 ADS 应用服务层

> ADS = Application Data Service，应用数据服务层

**一句话**：高度汇总、面向具体应用场景的"成品表"。

**典型例子**：
- 风控评分结果表（一行一个客户+评分）
- 大客户 Top100 榜单
- 监管报送报表数据

**工作中你要知道**：
**你开发的大部分数据服务 API 直接查 ADS 层**，因为这里的数据已经是"开箱即用"的了。

---

## 1.5 四层对比速查表

| 层级 | 中文名 | 数据粒度 | 数据量 | API 常用？ | 查一次速度 |
|------|--------|---------|--------|-----------|-----------|
| ODS | 原始层 | 最细 | 最大 | ❌ 不常用 | 最慢 |
| DWD | 明细层 | 明细 | 大 | ⚠️ 偶尔 | 慢 |
| DWS | 汇总层 | 轻度汇总 | 中 | ✅ 常用 | 快 |
| ADS | 应用层 | 高度汇总 | 小 | ✅✅ 最常用 | 最快 |

# 📗 Part 2：数据治理

## 2.1 四大治理域

```
         ┌──────────┐
         │  数据标准  │ ← 字段叫什么、类型是什么、口径是什么
         ├──────────┤
         │  数据质量  │ ← 数据准不准、全不全、及不及时
         ├──────────┤
         │   元数据   │ ← 数据的"数据"：谁创建的、什么时候、什么含义
         ├──────────┤
         │  数据血缘  │ ← 这个数据从哪来、经过了什么加工、被谁用
         └──────────┘
```

## 2.2 数据标准

**你开发时会遇到的情况**：

```sql
-- ❌ 不标准：同一个概念在不同表里叫不同名字
-- 表A：cust_id
-- 表B：customer_id
-- 表C：client_id

-- ✅ 标准后统一：全部叫 customer_id
```

**你的职责**：开发 API 时，如果发现字段不符合数据标准，主动提出修正。

## 2.3 数据质量

常见质量问题：
| 问题 | 例子 | 你的处理方式 |
|------|------|-------------|
| 缺失 | 客户手机号为空 | 接口文档明确标注该字段可能为空 |
| 重复 | 同一个客户两条记录 | 判断是否合法（不同时间版本），不合法则上报 |
| 不一致 | 性别有 M/F/0/1 | 按标准映射 |
| 不及时 | 当天数据还没到 | 设数据延迟告警 |

## 2.4 元数据

> 元数据 = 关于数据的数据

你接触到的元数据：
- 这张表有哪些字段（字段名、类型、长度、注释）
- 这个接口返回什么字段
- 这个指标的统计口径是什么

## 2.5 数据血缘

```
数据血缘追踪示例：

ODS.t_core_transaction
  │
  ├──→ DWD.t_trans_detail（经过清洗）
  │       │
  │       ├──→ DWS.t_cust_daily_sum（按客户+日期汇总）
  │       │       │
  │       │       └──→ ADS.t_cust_risk_score（风控评分）
  │       │               │
  │       │               └──→ API: /api/v1/risk/queryScore
  │       │
  │       └──→ ADS.t_txn_report（交易报表）
  │               │
  │               └──→ API: /api/v1/report/txnSummary
```

**你的价值**：当出问题时（比如风控评分不准），你能沿着血缘链追溯到是 DWD 还是 ODS 出了问题。

# 📙 Part 3：数据服务平台

## 3.1 平台四大职责

```
┌──────────────────────────────────┐
│          数据服务平台              │
│                                   │
│  1. 服务发布 ─ 把 API 注册到平台   │
│  2. 服务授权 ─ 谁能调这个 API？    │
│  3. 服务监控 ─ QPS/RT/错误率      │
│  4. 调用统计 ─ 谁调了多少次？      │
└──────────────────────────────────┘
```

## 3.2 你的日常操作

```
早上：
  1. 检查监控大盘：有没有接口报错/超时？
  2. 看调用统计：哪个业务方调用量异常？

接到需求：
  1. 需求评审（搞清楚取哪层数据、什么口径）
  2. 接口设计（URL、入参、出参）
  3. 开发 → 自测 → 提测

发版前：
  1. 接口回归测试
  2. 确认配置项（超时时间、限流阈值）

发版后：
  1. 观察监控
  2. 确认无异常告警
```
