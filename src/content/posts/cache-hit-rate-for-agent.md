## 核心理念

> **"Prompt caching is everything."** — 对长运行的 agent 产品（如 Claude Code），提示词缓存是最关键的性能杠杆。它让 API 往返可复用计算结果，大幅降低延迟和成本。

Claude Code 团队将 **缓存命中率（cache hit rate）** 作为关键运营指标，设置监控告警，命中率过低时声明 SEV（严重事件）。高缓存命中率不仅能降低成本，还能让订阅用户获得更慷慨的速率限制。

---

## 五个核心经验

### 1. 为缓存优化提示词结构

**原理：** 提示词缓存依赖 **前缀匹配**（prefix matching）——API 缓存从请求开头到每个 `cache_control` 断点之间的所有内容。顺序至关重要。

**Claude Code 的层级结构（从静态到动态）：**

**容易踩的坑（脆弱性）：**

- 在静态提示词中插入详细时间戳
    
- 工具定义以非确定性顺序排列
    
- 修改工具参数（如 Agent 可调用的子代理列表）
    

---

### 2. 用消息传递更新，而非修改提示词

> 当信息变陈旧时（时间变化、文件编辑），**不要直接修改系统提示词**——这会触发缓存失效。

**正确做法：** 通过 `<system-reminder>` 标签将更新数据注入下一轮的用户消息或工具结果中，既保留缓存，又让模型获取最新上下文。

---

### 3. 不要在会话中途切换模型

**反直觉的成本陷阱：** 如果用户已用 Opus 对话 100k token，想用 Haiku 回答一个简单问题——切换到 Haiku **反而更贵**，因为缓存对每个模型独立，切换后需要从头重建全部缓存。

**正确做法：** 用 **子代理（subagent）** 实现模型切换。例如 Opus 准备交接消息，Haiku 子代理执行后续操作。Claude Code 的 Explore 代理就大量使用此模式。

---

### 4. 绝不在会话中增删工具

> 这是最常见的缓存破坏方式之一。工具定义存储在缓存前缀中，任何修改都会使整个会话的缓存失效。

**Plan Mode 的缓存感知设计案例：**

|直觉做法（❌ 错误）|Claude Code 做法（✅ 正确）|
|---|---|
|进入计划模式时替换为只读工具集|所有工具始终存在，用专用工具建模状态转换|
|会破坏前缀匹配，缓存全部失效|`EnterPlanMode` / `ExitPlanMode` 本身就是工具|

**工具搜索的延迟加载（defer_loading）：**

- 问题：MCP 工具可能很多，全量 schema 放入每个请求成本高
    
- 解决：发送仅含名称的轻量桩（stub），标记 `defer_loading: true`，模型需要时通过工具搜索机制获取完整 schema
    
- 关键：前缀保持稳定，因为相同的桩每次都按相同顺序出现
    

---

### 5. 压缩对话而不破坏缓存

**问题：** 上下文窗口填满时需压缩（compaction），将对话总结为新会话。但朴素做法是用不同的系统提示词和零工具调用总结 API —— 前缀从第一个 token 就不同，整个长对话全按未缓存计费。

**Claude Code 的做法（Cache-Safe Forking）：**

1. 使用 **与父会话完全相同** 的系统提示词、用户上下文、工具定义
    
2. 前置父会话的对话消息，追加压缩指令作为最后一条用户消息
    
3. API 视角：与父会话上次请求几乎完全相同 → 前缀命中缓存
    
4. 只有压缩指令本身产生新的 token 费用
    
5. 需要维护 **压缩缓冲区** 确保有足够上下文空间
    

> 基于这些经验，团队已将压缩功能直接内置到 API 中。

---

## 关键总结

|#|原则|一句话|
|---|---|---|
|1|前缀匹配|前缀任何位置的改动都会使之后的所有内容缓存失效|
|2|消息优于编辑|通过消息传递更新，不动系统提示词|
|3|不变工具/模型|用工具建模状态转换，用子代理切换模型|
|4|监控缓存命中率|像监控 uptime 一样监控，小百分比也影响巨大|
|5|Fork 共享前缀|副计算（压缩、总结）必须复用父会话的缓存参数|
