## 1. VibeCoding 到底是什么

2025 年 2 月提出的定义：不审查不理解，直接 Accept AI 生成的代码。而现在所面对的则是你对你 Accept 的代码是否负有主要责任，而不是把锅甩给 AI。

## 2. AI 越来越强，作为独立程序员的优势在哪？

- **复杂业务逻辑？** 交给 AI 足够的上下文，健全逻辑严密的提示词库。
- **安全代码？** 告诉它注意什么，告诉它写出带有输入校验和权限检查的代码。
- **跨模块调用？** 给出接口文档，生成正确的调用链路。
- **性能优化？** 让它关注性能，N+1 查询和内存泄漏。

**人的优势在于：AI 做什么都需要你先做对什么。**

---

### 优势 1：问题定义

**实际功能抽象化成不同的需求**

定义问题是最难的部分。加一个功能意味着程序需要实现功能背后真正的需求定义，是人的一句话变成可执行的规范，AI 才能在框架内完成所需的代码。反过来如果自己都没想清楚具体的痛点，你的需求是模糊的，AI 给出的代码也一定是模糊的，给出一个"看起来像那么回事"，但是业务逻辑经不起推敲的输出。

**所以：** 个人的优势在于能把模糊的需求转化拆解为清晰的技术规范、明确的边界条件、异常处理、业务规范。

---

### 优势 2：上下文的解构能力

**AI 能否输出有质量的内容取决于你喂给 AI 的内容质量。**

上下文解构能力包含：

1. **知道给什么** — 哪些是需要让 AI 知道的，哪些是冗余
2. **知道不给什么** — 将项目整体塞给 AI，浪费且干扰模型判断
3. **知道如何组织** — 先给背景再给需求，还是先给需求再提环境，所获得的输出完全不同

**所以：** 个人能够，也必须能给出一套具有完整业务规则、相关代码片段和边界条件的 Prompt，而不是一句话就让 AI 写。AI 的输出上限取决于你的输入。

---

### 优势 3：结果的验证能力

**代码能跑起来 ≠ 代码正确**

AI 幻觉指的是大语言模型生成的内容看似合理、流畅、自信，但实际上是虚构、错误或与事实不符的现象。它不是"故意撒谎"，而是模型基于统计规律进行的概率性文本生成，本质上缺乏对真相的核实机制。

AI 代码经常看着对，跑得通，但是语义是错的。这些 bug 跑测试不一定能发现，如何验证是一种亟需加强的个人能力：

1. **业务语义验证** — 代码是否符合业务意图
2. **边界验证** — 极端情况下的行为是否符合预期
3. **回归验证** — 改动是否影响其他功能

一个完全不懂编程的人让 AI 写代码，和一位有经验的工程师使用 AI 辅助编程，差距不在于 AI 本身，而在于后者能快速识别不合理的设计、性能隐患和边界条件。

**所以：** AI 生成的代码我会重点验证业务语义，代码行为是否符合业务意图，人工理解一些测试可能覆盖不到的内容。对事实性结论进行交叉验证。

---

### 优势 4：技术决策能力

**最终拍板权在个人手里**

技术决策包括选型决策、架构决策、成本决策等。知道模型擅长什么（模式匹配、结构化输出、摘要、翻译）和不擅长什么（精确计算、实时信息、特定领域的专业知识），把任务放在合适的模型能力区间内。

与其把 AI 当作"给出最终答案的黑箱"，不如把它当作"输出初稿的协作者"。工作模式从 `AI 生成 → 直接使用` 变成 `AI 生成 → 人工审查 → 修改完善 → 最终产出`。这一步的深浅完全取决于后果的严重性——写一封不重要的邮件可以少审，部署到生产环境的代码必须多审。

**所以：** AI 能分析方案，能完善项目代码，但最终的决策是由我做的，因为决策不仅仅要考虑技术因素，还要结合团队现状、业务阶段、历史教训等综合完善。

---

### 优势 5：成本控制

**AI 编程不是免费的**

一次 Claude Code 的交互，可能消耗 5-20W 的 Token，一个完整项目跑下来可能费用超过了工程师一个月的工资。所以成本控制非常关键。

1. 知道什么时候使用能力更强的大模型，什么时候使用更快捷方便的小模型
2. 知道如何组织上下文能更节约 Token
3. 知道如何编写合理的 Prompt 才能减少来回的次数
4. 知道哪些任务让 AI 来做更合适，哪些任务自己来完成

**总结：**

> AI 做不了的潜台词是：等 AI 能做了，你也就没优势了。
> AI 需要你的潜台词是：只要 AI 还需要人工来驱动，我的优势就还在。

## 3. 四种具体策略

### 3.1 模型路由

**活用不同模型**

根据项目难度判断不同模型的使用，轻量化或主力模型降低 Token 成本。

### 3.2 上下文管理

**只给正确的代码**

用摘要代替全文；先让 AI 读取接口定义；需要细节时再补充；合理运用 Agent 辅助工具如 Skills 给模型提供专业领域知识和工作流程。

关键的一点：AI 读取到的无关代码越多，生成的质量越差。无关冗余信息会打消 AI 的"专注力"，这也是最近 AI 圈较火的**注意力工程（Attention Engineering）**——帮助并优化 AI 注意力，让它更专注于项目而不会被杂项干扰。

### 3.3 Prompt 优化

**注意你的提示词**

避免给出模糊的提示词，一次性说清楚而不是不断迭代你的需求。阐明你的需求、约束边界、上下文接口、数据库结构，或者给出较为明确的事例。

最近的 **Harness Engineering** 核心观点就是：AI 编程的重心正逐渐从 Prompt Engineering 转向 Context Engineering —— 不看你怎么说，而看你写入什么信息。

### 3.4 缓存和复用

**Cache 很重要，Memory 也很重要**

什么是缓存？Claude API 会缓存请求中**不常变化的部分**，比如：

- 系统提示（system prompt）
- 工具定义（tool schemas）
- 对话历史中较久远的部分

当相同内容再次发送时，Claude 直接从缓存读取而不是重新处理。缓存的内容写入一次后，在 5 分钟内有效。

| 术语 | 说明 |
|------|------|
| Cache Hit（缓存命中） | 某个 token 已经在缓存中存在，无需重新计算 |
| Cache Miss（缓存未命中） | 某个 token 不在缓存中，需要完整推理处理 |
| 命中率 | 缓存命中的 token 数 / 总 token 数 |

**为什么命中率重要？**

高命中率 = 更快、更便宜。在 Claude Code 中，ScheduleWakeup 工具会刻意避免 300 秒的睡眠间隔，因为缓存 TTL 是 5 分钟——睡超过 300 秒意味着缓存过期，下次请求全部 tokens 都要重新处理（0% 命中率）。

**Agent 复用**

当你启动一个子代理（通过 Agent 工具），它可以被继续使用而非每次都新建。通过 SendMessage 向已存在的代理发消息，可以复用它的完整上下文（它已经读过的文件、搜索结果等），避免重复加载。这相当于"热启动" vs "冷启动"。

**所以：** 只需要维护一套代码模板，AI 只需要根据具体需求填充差异部分，而不是每次都从 0 开始。配合 Prompt Engineering，相似任务的 Token 消耗将会大幅降低。

## 总结

1. **AI 越来越强，个人的优势是什么** — 优势不是写得比 AI 好，而是让 AI 写得更好。
2. **你负责的项目哪些要让 AI 写** — 代价小、重复度高，让 AI 加速。
3. **你怎么审查 AI 代码** — 业务语义；安全风险；工程性。特别注意 AI 合理但错误的代码——既能跑通但与业务需求有偏差、不一致。
4. **AI 代码上线后出了问题怎么处理** — 三步走：先止血 → 再定因 → 补流程。首先回滚降级，恢复线上功能；再查看监控日志确认影响范围和问题代码链路；想清楚为什么审查时没有发现问题；最后补充测试用例，加强审查重点，复盘。
