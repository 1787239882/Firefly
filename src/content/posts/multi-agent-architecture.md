## 核心问题

> "为什么你这里要拆多个 Agent？一个 Agent 多挂几个 Tool 不行吗？"
> 多 Agent 不是 "更智能" 三个字能糊弄过去的。面试官追问：智能在哪里？通信怎么做？失败怎么兜底？成本和延迟怎么算？

---

## 一、主 Agent 与子 Agent

**核心区别：** 子 Agent 不是主 Agent 的"手"，而是有独立大脑的"协作者"。

独立大脑体现在三个地方：

| 维度 | 说明 |
|------|------|
| **独立上下文窗口** | 每个子 Agent 有自己的上下文，互不干扰 |
| **独立系统提示词** | 不同角色可以有不同的行为规范 |
| **独立工具权限** | 搜索 Agent 只能搜索，写作 Agent 只能写作 |

---

## 二、通信链路四步拆解

### 第一步：任务分解

| 策略 | 说明 |
|------|------|
| **按功能域拆** | 不同 Agent 负责不同专业领域（搜索、分析、写作） |
| **按执行步骤拆** | 先后步骤交给不同 Agent（先调研→再分析→最后输出） |
| **按专业能力拆** | 根据擅长领域分配（代码→编程 Agent，数据→分析 Agent） |

> 任务拆分的本质：把**强耦合步骤**留在一个 Agent 里，把**可独立完成、可并行、上下文互相干扰**的部分拆出去。

### 第二步：上下文传递

主 Agent 传给子 Agent 的是一个"任务包"：

```json
{
  "task_id": "research_pricing_001",
  "role": "search_agent",
  "goal": "调研竞品A的定价策略",
  "input_context": ["竞品A公司名", "产品线", "需要对比的价格维度"],
  "allowed_tools": ["web_search", "read_webpage"],
  "output_schema": {
    "summary": "string",
    "evidence": "array",
    "confidence": "number",
    "risks": "array"
  },
  "deadline_ms": 30000
}
```

关键：传太少 → 子 Agent 不理解任务；传太多 → 浪费 token 还引入噪声。

### 第三步：子 Agent 独立执行

子 Agent 拿到任务后，自己决定执行路径：先调什么工具、信息不够怎么办、怎么重试。主 Agent 一般不干预。

### 第四步：结果回传与汇总

回传内容不只有结果，还包含：

- **执行摘要** — 做了什么、怎么做的
- **证据来源** — 用了哪些网页/文档/工具
- **置信度/状态** — 是否确信、是否部分失败
- **失败原因** — 哪个环节出问题

主 Agent 汇总需处理：结果冲突、格式统一、质量过滤、证据校验。

## 三、四种编排模式

| 模式 | 说明 | 适用场景 | 风险 |
|------|------|----------|------|
| **顺序管道** | 前一个输出是后一个输入 | 步骤有严格先后依赖 | 串行执行，总延迟 = 各 Agent 之和 |
| **Map-Reduce** | 并行派发，全部完成后汇总 | 子任务无依赖、可并行 | Reduce 写太弱 → 输出像拼接作文 |
| **层级嵌套** | 子 Agent 继续往下拆，树状结构 | 任务层次深、子任务复杂 | 层数越多，通信和调试难度指数上升 |
| **路由分发** | 主 Agent 判断类型分给对应专家 Agent | 请求类型多样 | 路由逻辑本身成瓶颈 |

## 四、Tool 调用 vs 多 Agent 架构

| 维度 | Tool 调用 | 多 Agent 架构 |
|------|-----------|---------------|
| **执行能力** | 单次调用返回结果 | 独立规划，多步推理 |
| **上下文** | 结果回到主 Agent 窗口 | 子 Agent 独立上下文 |
| **自主性** | 被动执行 | 主动规划、调整策略 |
| **并行性** | 由外层代码决定 | 天然适合并行拆分 |
| **容错** | 需主 Agent 额外处理 | 可按子任务重试/降级 |
| **协调开销** | 低 | 高（通信、汇总、冲突处理） |

---

## 五、Agent-as-Tool：中间态

把 Agent 包装成 Tool。主 Agent 通过 Function Calling 调一个"工具"，内部是完整 Agent。兼顾 Tool 的简单性和 Agent 的自主性。

**代表：** Claude Code — 主 Agent 通过 Task 工具调子 Agent，子 Agent 独立执行后返回结果。

**局限：** 主 Agent 对子 Agent 控制力弱，跑偏了无法中途纠正。

> Agent-as-Tool 的关键，不是把 Agent 随便包一层，而是要把输入边界、工具权限、输出 schema、超时和预算限制好。

## 六、场景决策树

### Tool 就够了

- 任务简单、步骤固定（查天气、发邮件）
- 子任务之间强依赖（无并行空间）
- 对延迟敏感
- 结果要求强一致（金额计算、权限校验）

### 需要多 Agent

- 任务复杂、需要独立规划
- 上下文会互相干扰
- 需要并行加速
- 需要专业化分工
- 需要隔离风险（搜索、代码执行拆到不同 Agent）

## 七、多 Agent 的代价

| 代价 | 说明 |
|------|------|
| **成本和延迟** | 三个子 Agent 并行，API 调用量可能是单 Agent 的三倍 |
| **协调复杂度** | 分解质量、传递粒度、冲突处理都是额外复杂度 |
| **可观测性差** | 跨多个 Agent 日志追踪，调试难度高一个量级 |
| **一致性风险** | 多个 Agent 结论不一致，汇总处理不好输出自相矛盾 |

一致性风险的解法不是"相信多数 Agent"，而是做**证据优先级**：
> 权威数据源 > 网页摘要，结构化数据库 > 模型推断，最新版本 > 历史版本。

## 八、真实框架对比

| 框架 | 通信方式 | 适合场景 | 主要风险 |
|------|----------|----------|----------|
| **CrewAI** | 任务链 / Manager 分配 | 流程明确、角色清晰 | Manager 质量决定上限 |
| **AutoGen** | Agent 之间多轮对话 | 需要讨论、评审、协商 | 对话轮次失控，成本和延迟上升 |
| **Claude Code** | 主 Agent 通过 Task 调用子 Agent | 复杂代码任务、上下文隔离 | 子 Agent 跑偏时中途控制弱 |

## 九、面试回答思路

1. 先讲通信链路四步：任务分解 → 上下文传递 → 独立执行 → 结果回传
2. 再讲为什么用多 Agent 而不用 Tool：上下文隔离、控制权下放、并行和专业化、容错粒度
3. 最后讲代价：协调复杂度、成本、调试难度、一致性风险
4. 补充判断标准：需要独立规划、上下文会互相干扰、需要并行加速时才上多 Agent

> 这个回答从通信机制到架构取舍，再到"不是所有场景都要多 Agent"的判断，比只背"Agent 比 Tool 智能"高一档。
