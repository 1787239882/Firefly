## 核心理念

> **TUN 模式工作在第三层（网络层），只处理 IP 数据包，不涉及数据链路层。** 它通过在操作系统内核中创建虚拟网络接口，将用户态程序的流量与内核网络栈桥接，实现高效的 IP 级隧道化。TUN 与 TAP（二层）的选择本质上是"效率 vs 能力"的权衡。

TUN（网络层隧道）和 TAP（数据链路层隧道）是两种主要的隧道模式，区别在于数据包处理的 OSI 层级。TUN 处理 IP 数据包（第三层），TAP 处理数据帧（第二层）。

---

## TUN 模式的优点

### 效率较高
仅处理 IP 层数据包，数据处理量小于 TAP 模式。只需封装 IP 包而非以太网帧，数据量小，传输效率高。在带宽受限的场景下优势尤为明显。

### 更强的跨平台兼容性
Linux、macOS、Windows、FreeBSD、OpenBSD、Android 等主流操作系统均原生支持 TUN 设备，便于在不同平台间建立一致的 VPN 连接。

### 适合路由流量
基于 IP 地址路由特定流量，适合通过配置路由表控制哪些流量经过 VPN 隧道。典型用例：远程访问特定内网资源（如只路由 `10.0.0.0/8` 到 VPN，其他流量直连互联网）。

### 节省网络资源
省略二层头部信息，对频繁传输大量数据的场景减少带宽占用。以典型 MTU 1500 为例，TUN 单包载荷比 TAP 多约 14-18 字节（以太网头大小），累积收益可观。

### 简单性和稳定性
不涉及链路层，VPN 管理和设置更简单，连接更稳定，对操作系统底层依赖少，便于调试和排错。

## TUN 模式的缺点

### 不支持非 IP 协议
无法处理 ARP 请求、DHCP 广播、NetBIOS 等非 IP 协议数据包。如果隧道中需要传输这些，TUN 模式无法满足。

### 无法支持二层广播和组播
在三层工作意味着不能传输广播/组播包。依赖二层通信的功能（ARP 解析、局域网文件共享、设备发现等）在 TUN 模式下不可用。

### 有限的局域网模拟能力
TUN 的设计目标是远程访问而非二层桥接。需要模拟完整局域网时（远程控制、网络扫描、LLMNR/mDNS 发现等），TUN 不如 TAP 适用。

### 路由配置较复杂
流量分流需要精细的路由表配置，尤其多网段场景。对不熟悉路由规则的用户配置门槛较高，错误配置可能导致流量泄露。

### IP 层安全风险
IP 协议本身存在已知攻击面（IP 欺骗、分片攻击、ICMP 隧道等）。高安全需求场景下需配合高强度加密算法，否则可能受 IP 层攻击。

## 补充辨析：TUN vs TAP 的核心差异

| 维度 | TUN（三层） | TAP（二层） |
|------|------------|------------|
| OSI 层级 | 第三层（网络层） | 第二层（数据链路层） |
| 数据单元 | IP 数据包 | 以太网帧 |
| 协议支持 | 仅 IP | 所有以太网协议（ARP、IPv4、IPv6、MPLS 等） |
| 广播支持 | ❌ | ✅ |
| 效率 | 较高（头开销小） | 较低（完整的二层头部） |
| 适用场景 | 远程访问、站点互联 | 局域网桥接、虚拟机网络 |
| 配置复杂度 | 路由表配置 | 网桥/交换机配置 |

## 常见场景与最佳实践

### 适用 TUN
- 远程办公连接公司内网（仅路由内网网段）
- 分站点间 IP 互联（站点到站点 VPN）
- 移动设备 VPN 客户端（带宽和电量敏感）
- DNS 或指定域名分流（按 IP 段控制）

### 应考虑 TAP
- 需要 ARP 广播（如 Windows 网络发现、文件共享）
- 二层协议依赖应用（DHCP 中继、某些游戏协议）
- 虚拟机/容器网络桥接
- 需要完整局域网体验的远程桌面场景

### 混合方案
某些 VPN 实现支持同时创建 TUN 和 TAP 接口，分别处理 IP 流量和非 IP 流量。WireGuard（纯 TUN）通过集成方案弥补部分二层能力缺失。

## 关键总结

| # | 要点 | 一句话 |
|---|------|--------|
| 1 | 三层 vs 二层 | TUN 是 IP 层隧道，TAP 是以太网帧隧道 |
| 2 | 效率优先 | TUN 数据包更小，带宽和 CPU 效率更高 |
| 3 | 兼容性 | TUN 几乎所有操作系统原生支持，TAP 部分平台需额外驱动 |
| 4 | 路由能力 | TUN 适合精细化路由控制，TAP 适合完整局域网体验 |
| 5 | 协议限制 | TUN 不能传输非 IP 协议，TAP 无此限制 |
